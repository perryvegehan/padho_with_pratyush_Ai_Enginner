# System Role and Temperature in LLMs

## Message Roles in an LLM

Messages sent to an LLM have a **role** and **content**.

The three common roles are:

### 1. User Role

Represents the person interacting with the LLM.

```python
{
    "role": "user",
    "content": "What is Machine Learning?"
}
```

### 2. Assistant Role

Represents the response generated by the LLM.

### 3. System Role

Provides instructions that define **how the LLM should behave** or **what role/persona it should take** during the conversation.

---

# System Role

## Definition

A **System Role** is a high-level instruction given to an LLM that defines its role, behavior, perspective, or relationship with the user.

It answers the question:

> **"Who should the LLM behave as?"**

The same user message can produce different responses depending on the system instruction.

---

## Example: Same Message, Different System Role

### User Message

```text
I love you.
```

### System Role 1

```text
You are my loving girlfriend.
```

Possible response:

```text
I love you too.
```

### System Role 2

```text
You are my strict office colleague who also happens to be my manager.
```

Possible response:

```text
This kind of communication is not appropriate in a professional workplace.
```

The **user message remains the same**, but the response changes because the **LLM's assigned role changes**.

---

## System Role in Code

```python
message_system = {
    "role": "system",
    "content": "You are my loving girlfriend."
}

message = {
    "role": "user",
    "content": "I love you baby."
}

messages = [
    message_system,
    message
]
```

The system message should come **before** the user message:

```text
System Instruction
        ↓
User Message
        ↓
LLM Response Based on the System Instruction
```

# Real-World Use of System Roles

A company may use the same underlying LLM to create multiple AI agents with different responsibilities.

For example:

```text
Same LLM
   │
   ├── Senior Architect
   ├── Junior Developer
   └── Tester
```

### Senior Architect

```text
You are a senior-level software architect.
```

The LLM may focus on:

- System design
- Security
- Maintainability
- Reliability
- Overall architecture

### Junior Developer

```text
You are a junior developer.
```

The LLM may focus more on:

- Writing code
- Fixing bugs
- Implementation details

### Tester

```text
You are a software tester.
```

The LLM may focus on:

- Finding bugs
- Testing behavior
- Edge cases

## Important Point

> **System Role changes the role or perspective from which the LLM responds.**

This allows the same underlying LLM to behave like different specialized AI agents.

# Temperature

## Definition

**Temperature** is a parameter that controls the degree of **randomness and creativity** in an LLM's output.

It answers the question:

> **"How predictable or creative should the response be?"**

In the API demonstrated in the lecture, the temperature range used was:

```text
0 → Lower randomness
1 → More creativity
2 → Higher randomness
```

> The valid temperature range depends on the model and API being used.

---

# How Temperature Affects Output

LLMs generate responses by predicting likely next tokens.

Temperature influences how conservative or varied that prediction process is.

### Low Temperature

```text
Temperature = 0
```

The model generally favors more likely and predictable outputs.

```text
More Predictable
Less Random
Less Creative
```

### Medium Temperature

```text
Temperature = 1
```

The model allows more variation.

```text
Balanced
More Variety
More Creative
```

### High Temperature

```text
Temperature = 2
```

The model allows greater randomness in its output.

```text
Less Predictable
More Random
More Creative
```

---

# Simple Example

Suppose we ask:

```text
Sunday, Monday, ...
```

A low-temperature answer is likely to be:

```text
Tuesday
```

because it is the most obvious continuation.

With more randomness, the model may consider less obvious alternatives instead of always choosing the most predictable continuation.

The lecture uses this example to explain that:

> **Temperature modifies the randomness of the model's predictions.**

# Real-World Use of Temperature

Different applications require different levels of randomness.

## Low Temperature Use Case

For tasks where consistency and predictability are important:

```text
Low Temperature
        ↓
More Consistent and Predictable Responses
```

Like **medical/doctor AI agent** as an example of a use case where unnecessary creativity would be undesirable.

---

## High Temperature Use Case

For creative tasks such as:

- Story generation
- Brand-name generation
- Creative writing
- Brainstorming

Example:

```text
High Temperature
        ↓
More Variation and Creativity
```

A storytelling AI may benefit from a higher temperature because different and imaginative outputs are useful.

---

# System Role vs Temperature

| System Role | Temperature |
|---|---|
| Defines **who the LLM should behave as** | Controls **how random or creative the output is** |
| Changes the LLM's role or perspective | Changes the variation in prediction |
| Example: Architect, Developer, Tester | Example: Low, Medium, High randomness |
| Used to specialize AI agents | Adjusted according to the use case |

# Complete Code Example

```python
import os
from dotenv import load_dotenv
from groq import Groq

load_dotenv()

api_key = os.getenv("GROQ_API_KEY")

if not api_key:
    raise ValueError("GROQ_API_KEY is missing")

client = Groq(api_key=api_key)

model = "llama-3.3-70b-versatile"

message_system = {
    "role": "system",
    "content": "You are a brand manager who suggests names for my brand. Suggest one name only."
}

message = {
    "role": "user",
    "content": "Suggest a name for my food company."
}

messages = [
    message_system,
    message
]

response = client.chat.completions.create(
    model=model,
    messages=messages,
    temperature=1
)

answer = response.choices[0].message.content

print(answer)
```

# Important Points

- An LLM message contains a **role** and **content**.
- The three common roles are `system`, `user`, and `assistant`.
- The **system role** defines the LLM's behavior, role, or perspective.
- The same user prompt can produce different responses under different system instructions.
- **Temperature** controls randomness and variation in generated output.
- Lower temperature generally produces more predictable responses.
- Higher temperature generally produces more varied and creative responses.
- The correct temperature depends on the use case.
- The exact supported temperature range depends on the model/API.
- LLM outputs are not completely predictable, so the same prompt may not always produce exactly the same response.

# Quick Questions

## What is a System Role in an LLM?

A system role is a high-level instruction that defines how an LLM should behave or what perspective it should use while responding.

## What is Temperature in an LLM?

Temperature is a generation parameter that controls the randomness and variation of the model's output.

## What is the difference between System Role and Temperature?

The system role defines **who or what the LLM should behave as**, while temperature controls **how predictable or creative its generated response should be**.

## Why are different System Roles useful?

They allow the same underlying LLM to be used for different specialized tasks, such as an architect, developer, or tester.

## When should we use a lower temperature?

A lower temperature is useful when more consistent and predictable responses are preferred.

## When should we use a higher temperature?

A higher temperature is useful for tasks where variety and creativity are valuable, such as storytelling, brainstorming, and brand-name generation.
